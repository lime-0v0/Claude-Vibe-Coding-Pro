## 읽는 순서

```
00-overview  →  01-product  →  02-specs  →  03-design  →  04-tasks  →  05-conventions
  (전체 맥락)      (무엇을)       (어떻게)      (어떻게 보여줄)   (언제/무엇부터)   (어떤 규칙으로)
```

**새 작업을 시작하기 전, 이 순서대로 전부 읽는다.**
문서를 건너뛰면 이미 결정된 사항을 모른 채 작업하게 된다.

## 왜 이렇게 나누는가 — 관심사 분리(Separation of Concerns) 원칙

소프트웨어 설계에서 각 모듈이 하나의 책임만 지듯,
문서도 **하나의 파일이 하나의 관심사**만 다뤄야 한다.

| 문제 상황 | 관심사 분리 적용 후 |
|-----------|-------------------|
| 기획이 바뀌어도 기술 명세까지 전부 고쳐야 한다 | `01-product` 만 수정하면 된다 |
| 디자인 변경이 작업 계획과 섞여 있어 혼란스럽다 | `03-design` ↔ `04-tasks` 가 독립적으로 관리된다 |
| 컨벤션 규칙을 찾으려면 전체 문서를 뒤져야 한다 | `05-conventions` 한 파일만 열면 된다 |

**핵심 원칙:**
- 각 파일은 **변경 이유가 하나**여야 한다 (단일 책임)
- 파일 간 **의존 방향은 위에서 아래** (`00` → `05`) 로만 흐른다
- 상위 파일(`00`, `01`)은 자주 읽히고, 하위 파일(`04`, `05`)은 자주 수정된다
