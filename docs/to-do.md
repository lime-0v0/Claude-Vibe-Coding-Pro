# to-do.md — 기능 후보 목록

> MVP 이후 도입을 검토할 기능들을 세 관점에서 정리한다.  
> 우선순위는 `🔴 높음 / 🟡 중간 / 🟢 낮음`으로 표기한다.  
> 구현 결정은 이 문서를 참고해 별도 설계 문서(02-specs 개정 또는 신규)를 작성한 뒤 진행한다.

---

## 1. 개발자 관점 — 코드 품질·인프라·확장성

현재 쌓인 기술 부채와 확장 시 병목이 될 지점을 해소하는 항목들이다.

| 우선순위 | 기능 | 현재 문제 | 기대 효과 |
|---------|------|-----------|-----------|
| 🔴 | **에러 처리 개선** — `alert()` 제거 → 토스트 알림 UI | 브라우저 `alert()`는 UX 최악, 모달도 막힘 | 사용자 경험 일관성 확보 |
| 🔴 | **WebSocket 실시간 업데이트** — 3초 폴링 제거 | 폴링은 불필요한 네트워크 요청을 지속 발생 | 서버 부하 감소, 즉각 반영 |
| 🔴 | **JWT 인증** — 로그인/로그아웃, 토큰 기반 세션 | 현재 인증 없음 → 누구나 전체 데이터 조회·수정 가능 | 팀별 데이터 격리의 전제 조건 |
| 🟡 | **DB 마이그레이션** — SQLite → PostgreSQL (Alembic) | SQLite는 동시성 한계, 파일 기반이라 배포 불안정 | 프로덕션 수준 안정성 확보 |
| 🟡 | **CI/CD 파이프라인** — GitHub Actions, push 시 pytest 자동 실행 | 현재 테스트는 수동 실행 | 회귀 버그 조기 감지 |
| 🟡 | **Docker 컨테이너화** — backend + frontend 이미지 분리 | 환경 의존성 수동 관리 | 어디서나 동일 환경 재현 |
| 🟡 | **로깅 시스템** — `print` 제거 → `logging` 모듈, 요청/에러 로그 파일 기록 | 현재 에러 추적 불가 | 장애 원인 분석 가능 |
| 🟢 | **프론트엔드 번들링** — Vite 도입, Tailwind CDN → 로컬 빌드 | CDN 의존 → 오프라인 불가, 트리쉐이킹 없음 | 번들 크기 최적화, 오프라인 지원 |
| 🟢 | **API 버전 관리** — `/api/v1/tasks` prefix 도입 | 현재 버전 없음 → 브레이킹 체인지 시 클라이언트 일괄 수정 필요 | 하위 호환 유지 가능 |
| 🟢 | **테스트 커버리지 확대** — 프론트엔드 E2E 테스트 (Playwright) 자동화 | 현재 pytest 17개는 백엔드만 커버 | 전체 CRUD 시나리오 회귀 방지 |

## 2. 사용자(실제 유저) 관점 — 업무를 직접 처리하는 팀원

매일 앱을 열고 자기 업무를 관리하는 사람의 불편함을 제거하는 항목들이다.

| 우선순위 | 기능 | 사용자 불편 | 기대 효과 |
|---------|------|------------|-----------|
| 🔴 | **담당자(assignee) 지정** — 업무에 팀원 이름 연결, "내 업무만 보기" 필터 | 전체 목록에 내 업무와 남의 업무가 섞임 | "지금 내가 뭐 해야 하지?"에 바로 답 |
| 🔴 | **업무 검색** — 제목 키워드 검색, 실시간 필터링 | 업무가 많아지면 스크롤로 찾아야 함 | 빠른 탐색 |
| 🔴 | **마감 임박 알림** — 브라우저 Notification API, D-1 / D-0 기준 | 마감을 놓쳐도 앱을 안 보면 모름 | 마감 초과 감소 |
| 🟡 | **우선순위 필드** — `low / medium / high / urgent` 4단계, 배지 표시 | 현재 상태(todo/in_progress/done)만으로 긴급도 구분 불가 | 무엇을 먼저 해야 하는지 명확 |
| 🟡 | **업무 정렬** — 마감일순 / 생성일순 / 우선순위순 / 상태순 | 현재 생성 역순 고정 | 상황에 맞는 뷰 선택 가능 |
| 🟡 | **체크리스트(하위 업무)** — 업무 안에 소항목 추가, 체크 완료 | 큰 업무를 단계별로 쪼갤 수 없음 | 업무 진행률 가시화 |
| 🟡 | **태그/레이블** — 자유 태그 부여, 태그별 필터 | 프로젝트·카테고리 구분이 상태 3개뿐 | 다차원 분류 가능 |
| 🟡 | **Kanban 보드 뷰** — todo / in_progress / done 컬럼, 드래그로 상태 변경 | 현재 리스트 뷰 하나뿐 | 시각적 흐름 파악, 드래그 UX |
| 🟢 | **업무 복제** — 기존 업무를 템플릿으로 새 업무 생성 | 반복 업무를 매번 새로 입력 | 반복 입력 비용 감소 |
| 🟢 | **설명 Markdown 지원** — description 필드에 마크다운 렌더링 | 현재 plain text만 지원 | 링크·코드·체크리스트 작성 가능 |
| 🟢 | **완료 업무 아카이빙** — done 상태 업무를 별도 뷰로 분리 | 완료 업무가 목록을 채워 현재 업무 파악을 방해 | 목록이 깔끔하게 유지됨 |

## 3. 사용자(관리직) 관점 — 팀 현황을 파악하는 팀 리더

"지금 누가 뭐 해?"를 해결하는 것이 제품 목표다. 관리직의 시각에서 남은 미충족 니즈를 정리한다.

| 우선순위 | 기능 | 관리자 니즈 | 기대 효과 |
|---------|------|------------|-----------|
| 🔴 | **팀원별 업무 현황 대시보드** — 팀원 이름별 todo/in_progress/done 카운트 한 화면 표시 | 현재 개인별 업무량·진행률 파악 불가 | 슬랙으로 물어보지 않아도 됨 |
| 🔴 | **마감 초과 업무 하이라이트** — 전체 뷰에서 D+N 업무 별도 섹션 또는 색상 강조 | 마감 초과 업무가 목록에 섞여 눈에 안 띔 | 즉각 조치 가능 |
| 🔴 | **팀/프로젝트 단위 분류** — 업무에 프로젝트 연결, 프로젝트별 필터·진행률 | 현재 모든 업무가 한 목록에 섞임 | 프로젝트별 현황 파악 가능 |
| 🟡 | **캘린더 뷰** — 마감일 기준 월간/주간 캘린더, 업무 일정 시각화 | 리스트로는 일정 밀집도 파악이 어려움 | 리소스 충돌·과부하 사전 감지 |
| 🟡 | **워크로드 뷰** — 팀원별 진행 중 업무 수·마감 분포 시각화 (막대 그래프 등) | 특정 팀원에게 업무가 몰려도 모름 | 균형 배분, 번아웃 예방 |
| 🟡 | **업무 변경 이력** — 상태 전이·수정 내역 타임라인 기록 | 업무가 언제 시작되고 얼마나 걸렸는지 추적 불가 | 사후 분석·리뷰 근거 확보 |
| 🟡 | **통계 리포트** — 완료율, 평균 처리 시간, 마감 준수율 (주간/월간) | 팀 생산성을 수치로 확인할 방법이 없음 | 정기 회의 자료, KPI 측정 |
| 🟢 | **데이터 내보내기** — 업무 목록 CSV/Excel 다운로드 | 외부 보고 시 수동 정리 필요 | 보고서 작성 비용 감소 |
| 🟢 | **슬랙/Teams 알림 연동** — 업무 생성·완료 시 채널 알림 | 앱을 따로 열어야만 현황 확인 가능 | 기존 커뮤니케이션 채널에서 인지 |
| 🟢 | **권한 관리** — 팀원 초대/제거, 역할별 접근 제어 (admin/member) | 현재 인증 없음 → 팀 단위 데이터 격리 불가 | 보안, 멀티팀 운영 기반 |

## 우선 착수 추천 순서

위 항목 중 **의존 관계**와 **효과 대비 비용**을 고려한 권장 순서다.

```
1단계 (기반 작업)
  └─ JWT 인증           ← 담당자·팀·권한 모든 기능의 전제 조건
  └─ 에러 처리 개선     ← 현재 UX 결함, 비용 낮음

2단계 (핵심 사용성)
  └─ 담당자 지정        ← 1단계 인증 완료 후 가능
  └─ 우선순위 필드      ← 독립 기능, 비용 낮음
  └─ 업무 검색          ← 독립 기능, 비용 낮음

3단계 (시각화)
  └─ 팀원 현황 대시보드  ← 2단계 담당자 기능 완료 후 가능
  └─ Kanban 보드 뷰     ← 독립 기능
  └─ 마감 임박 알림     ← 독립 기능

4단계 (인프라)
  └─ WebSocket 실시간   ← 폴링 제거
  └─ DB → PostgreSQL    ← 배포 안정화
  └─ CI/CD              ← 개발 프로세스 자동화
```
